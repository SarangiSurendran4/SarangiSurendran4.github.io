Documentation Automation  ·  API Integration  ·  AI Workflow

Building a self-healing documentation maintenance system

Problem

The team was spending hours each week on manual documentation checks: broken links buried across 500+ articles, outdated screenshots that no longer matched the live product UI, and recurring article hygiene tasks with no scalable solution. The built-in Document360 broken link checker was a paid add-on with usage limits the team didn't want to invest in.

What I built

Two automated workflows using Python, the Document360 API, and Claude Code. The first crawls all articles, detects broken links, logs each one with article name, URL, anchor text, and reason for failure into a CSV, then uses the API to auto-heal fixable links and republish the article. The second compares live screenshots in articles against the current product UI and flags visual drift before it reaches users.

Impact

Both tools were adopted team-wide. Manual link checks were eliminated entirely. Screenshot drift is now caught before releases rather than after user complaints. The team reclaimed time previously lost to repetitive maintenance work and redirected it toward higher-quality content creation.

Replaced a paid tool. Eliminated manual link checks across 500+ articles. Screenshot quality improved measurably before releases.

AI-Assisted Analysis  ·  Support Operations  ·  DAP Strategy

Using AI to turn 100+ support tickets into a targeted content rollout

Problem

Support tickets were being resolved individually without surfacing patterns. There was no systematic way to identify which issues were recurring, which features users consistently misunderstood, or where documentation gaps were causing the most friction.

What I did

Took a bulk export of support tickets and used Claude to identify recurring themes, cluster related issues, and surface the top friction points. Used those findings to plan a targeted rollout of DAP content: flows, smart tips, and launchers placed at the exact points in the product where users were getting stuck.

Outcome

Recurring support tickets on the targeted features decreased after the content rollout. The analysis took hours instead of weeks and gave the team a replicable approach for future content prioritization decisions.

Reduced recurring support tickets on targeted features. Replicable AI-assisted analysis workflow established.

How I Work

How I approach documentation

Good documentation is a research problem before it is a writing problem. Here is how I think about it.

Understand before writing

I start with support tickets, customer calls, and hands-on product testing before writing anything. The goal is to understand what users are actually trying to do, where they get confused, and what they need to hear, not just what the feature does.

Structure around user behavior

I organize content around what users are trying to accomplish, not around how the system is built. This often means restructuring feature guides as task flows, leading with troubleshooting scenarios, and cross-linking related content where users actually need it.

Test and validate content

I work with PMs, engineers, and digital adoption consultants to review documentation after it is written. If something is unclear or misleading, I want to find that out before it reaches users, not through a support ticket afterwards.

Build for maintenance, not just launch

Documentation that is accurate at launch and broken six months later is not finished work. I build automation workflows that catch outdated content, broken links, and UI drift so the team is not manually auditing hundreds of articles each cycle.

Use AI purposefully

I use AI tools for first drafts, support ticket analysis, content simplification, peer review prompts, and reusable documentation workflows. Every AI output gets validated before use. The goal is faster iteration on high-quality work, not volume for its own sake.

Work across the team

The best documentation decisions come from conversations with PMs, engineers, support teams, and customers. I attend product calls, listen to customer recordings, and share findings back with the team to improve both content and process.
